     Recommended approach

     1) 增加“结构化文献检索层”，在 evidence 获取时优先走该层

     Integration point

     evidence(...) 作为主入口。

       - WebSearcher.search_structured_literature_and_fetch(...)

     Why here

     - 不改变 external_agent 的证据契约（仍产出 EvidenceBundle / WebSearchResult
      风格）。
     - structured path 只在“论文/预印本”场景触发，符合替换范围要求。

     采用“启发式 + 可开关”组合：
     - AGDEBUGGER_LITERATURE_ENABLED=1 时启用
     - 且满足以下任一：
       - claim/query 包含明显论文标识（PMID/PMCID/DOI/arXiv/biorxiv/medrxiv）
       - 或 strategy 是科学概念/科学事实类（例如 scientific_concept_discovery）

     （避免所有 evidence 都走结构化导致覆盖面下降。）

     2) 在 agdebugger/websearch 内实现本地 clients（参照参考实现）

     不依赖 ToolUniverse/MCP，直接在 agdebugger/websearch 内实现轻量
     client，模式/字段尽量对齐参考实现。

     New files (必要的新文件)

       - 参照参考 PubMedClient：esearch → efetch(xml) → parse、elink 做
     PMID→PMCID

       - 参照参考 CrossrefClient：GET /works，做 DOI 归一化 + landing page
     - AgentDebug/agdebugger/websearch/biorxiv_client.py

     - identifiers: pmid, pmcid, doi
     - metadata: title, abstract, journal, year, authors
     - source fields: retrieval_source, landing_page_url, jats_xml_path,
     published_doi

     这样既能让 query 更像人写的论文检索式，又不至于把 structured retrieval
     的稳定性完全交给模型随机性。

     Recommended approach

     1) 在 structured retrieval 前新增一个“paper query agent”步骤

     Where

     - AgentDebug/agdebugger/external_agent/evidence_provider.py

     Why here

     WebSearchEvidenceProvider.build_evidence(...) 已经是 structured / web
     fallback 的总路由入口，而且它已经持有：
     - self._sampler（通过 OpenAIAdapterSampler 构造）
     - scientific claim 原文 / source_ref / original_statement

     因此最适合在这里新增：
     - _build_agent_structured_query(...)

     而不是把这层逻辑塞进 web_scraper.py 或 client 本身。

     2) Agent paper query 的职责要非常窄

     Goal

     让 agent 只做一件事：
     - 根据当前 scientific claim，输出一个紧凑、title-like 的 scholarly paper 
     query

     It should output

     建议输出一个极小 JSON：

     {
       "paper_query": "MCR-1 colistin resistance phosphoethanolamine 
     transferase"
     }

     Why keep schema minimal

     当前 structured retrieval 下游只需要一个字符串 query：
     - search_structured_literature_and_fetch(query, ...)

     因此 agent 不需要一开始就输出复杂对象（synonyms / filters / source
     ranking）。
     先把 schema 控制在最小范围内，能降低：
     - JSON 解析失败概率
     - latency
     - prompt 复杂度
     - 额外 failure modes

     3) Prompt 设计：明确是“paper query”，不是一般 web query

     Reuse

     可复用现有两类模式：
     - websearch/serper_client.py 中的 search-planning JSON prompt 风格
     - external_agent/llm.py / external_agent_controller.py 中的 “JSON only /
     one object only” 约束方式

     Prompt requirements

     agent prompt 应明确要求：

     - 输出适合 PubMed / Crossref 的论文检索式
     - query 应：
       - 短
       - 实体化
       - title-like
       - 仅保留 gene / protein / compound / disease / method / mechanism
     等核心词
     - query 不应包含：
       - answer
       - option
       - correct
       - incorrect
       - provided choices
       - claim 的长句式重述
       - 过多推理/解释性词语

       - 优先保留主实体 + 1 个机制/phenotype + 必要 partner
     - 只返回 JSON，不返回 prose

     - 好的 paper query：
       - MCR-1 colistin resistance phosphoethanolamine transferase
       - MCR-1 DsbA disulfide bond folding
     - 不好的 query：
       - MCR-1 DsbA-mediated Enhanced MCR- activity folding DsbA improves 
     activation...

     4) 不建议完全去掉 deterministic fallback

     Why

     从当前代码结构和 recent runs 看，完全依赖 agent query 有几个风险：

     1. 每个 claim 多一次 LLM 调用，明显增加 latency
     2. scientific concept discovery 当前本来就偏向 deterministic 路线，完全改成
      agent 会增加不稳定性
     3. agent 输出可能：
       - 太长
       - 跑题
       - 结构化 JSON 不合法
       - 被 claim 文本里的噪音词带偏

     因此 structured retrieval 推荐改成：

     1. 先调用 agent 生成 paper_query
     2. 若 agent query 合法且非空：

       - 回退到 deterministic _build_structured_literature_query(...)
     4. 若 deterministic structured 仍失败：
       - 再回退到原 web search

     这能兼顾：
     - 更聪明的 query
     - 保守的稳定性兜底

  ---
  真正的问题：replacement_text 是“整条 message 的新全文”

  看现在报错的 action，repair_source=concept_repair_fallback
  也还是失败，而且内容像这样：

  - Based on the Nature paper ...
  - Considering the options:
  - Option 1: ...
  - Option 2: ...

  这说明什么？

  _build_deterministic_concept_repair_action() 生成的不是“纯局部 patch”

  而是：

  - 把原 assistant message 整条取出来
  - 在其中某个 anchor 位置替换成 corrected_claim_text
  - 然后返回整条替换后的完整 message content

  相关代码就是：

  - _build_precise_replacement_text(...)
    - run_dataset_autodebug.py:1035-1083

  它的行为是：

  parts.append(original_content[cursor:repair["start"]])
  parts.append(repair["replacement"])
  parts.append(original_content[cursor:])
  replacement_text = "".join(parts)

  也就是说返回的是：

  “原整段消息 + 局部替换后的全文”

  ---
  为什么这会被判成 direct answer injection

  因为 validator 现在检查的是：

  - replacement_text/body 的全文

  而不是“这次新插入/替换的局部 span”。

  看这里：

  - _validate_concept_repair_action(...)
    - run_dataset_autodebug.py:822-861

  关键逻辑：

  if any(_contains_direct_answer_rewrite(text) for text in
  _action_payload_strings(action)):
      return "concept repair forbids direct answer injection in 
  replacement_text/body"

  而 _action_payload_strings(action) 会把整个 replacement_text 拿去检查。

  所以即使：

  - 你这次真正新补进去的 corrected_claim_text 是干净的

  但是如果原 message 本来后半段就带着
  - option 分析
  - 最终答案
  - <answer>
  - TERMINATE

  那么整个重建后的 replacement_text 仍然会包含这些旧内容。

  于是 validator 会误判成：

  ▎ “你这次 repair 注入了答案”

  但实际上是：

  ▎ 原消息里残留的答案文本还在，局部 patch 没把它切掉

  ---
  这就是现在 workflow 卡死的真正根因

  当前 concept repair 的冲突是：

  repair 机制

  想做的是：
  - 局部 span replacement
  - 保留上下文

  validator 机制

  实际检查的是：
  - 整个 replacement 后的全文不能含 answer-like text

  这两者在“原 assistant message 后半段已经带答案”的情况下，天然冲突。

  ---
  一句话总结

  当前不是 planner 单独有问题，而是：

  deterministic concept repair 会返回“整条消息的全文替换结果”，而 validator 
  又按“整条全文不能含答案”来检查，所以只要原消息后半段已经有 option/answer 
  痕迹，局部修复也会被误杀。

  ---
  现在最该怎么修

  最直接有两条路，且我更推荐第 1 条：

  方案 1：validator 只检查“新插入/替换的局部内容”，不要检查整条 replacement_text

  也就是：
  - concept repair 允许原 message 里保留未改动的旧文本
  - 只禁止 新增 patch span 里出现：
    - <answer>
    - TERMINATE
    - option selection language

  这是最符合“局部 patch”语义的。

  方案 2：做 patch 时顺便裁掉原 message 后面的答案段

  也就是：
  - 不只是替换错误 claim
  - 还把后续 option/answer 区域截断

  但这个会重新变回“重写式 repair”，风险更高。

  ---
  结论

  我现在已经定位到真正的机制冲突了：

  - evidence 已经不是主问题
  - planner 乱写是问题之一
  - 但更根本的是：
    - 局部 patch 返回整条全文
    - 全文 validator 把原残留答案也算成注入

  如果你同意，我下一步建议我直接修：

  修复方向

  让 strict concept repair validator 只校验“局部新补进去的 corrected_claim_text 
  / patch span”，而不是整条 replacement_text 全文。

  这应该才是把 workflow 真正打通的关键一步。
